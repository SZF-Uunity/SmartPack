# 前后端联调与代码功能审查报告

## 一、联调结论

本次对 SmartPack 项目执行了“前后端联调就绪性审查”，重点覆盖：

- 前端控制台到后端 API 的调用闭环是否完整。
- 默认角色权限是否满足实际联调链路。
- 本地启动后是否具备最小可用测试数据。

结论：

- 已补齐默认引导数据能力（用户、API Key、客户、材质、箱型、产品），新环境启动即可联调。
- 已修复权限模型中影响联调的断链问题（packer 读取 AI 接口权限）。
- 已收敛 API Key 管理权限到 sysadmin 语义，更符合企业治理标准。

## 二、发现的问题与修复

### 1) 联调“冷启动不可用”问题

**问题**：数据库为空时，前端即使能打开页面，也无法直接完成完整联调（没有账号、API Key、基础主数据）。

**修复**：新增 `BootstrapService`，启动时仅在空库写入最小演示数据并输出中文日志。

### 2) packer 角色调用 AI 解析被拒绝

**问题**：前端默认角色为 `packer`，但 `packer` 无 `catalog:read` 权限，会导致 `/ai/extract` 403。

**修复**：在 `ROLE_PERMISSIONS` 中为 `packer` 增加 `catalog:read`。

### 3) API Key 管理权限过宽

**问题**：创建 API Key 接口之前使用 `catalog:write` 控制，语义不准确且权限边界过宽。

**修复**：新增 `security:manage` 权限并用于 API Key 创建接口，符合企业安全职责划分。

## 三、联调建议

- 前端登录上下文可先使用默认 key：`dev-smartpack-key-please-change`（仅开发联调）。
- 生产环境必须关闭默认种子数据或通过环境变量禁用。
- 下一步建议增加“联调检查页”（一键探测健康、权限、基础数据完整性）。
